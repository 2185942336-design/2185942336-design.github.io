# CTF Basic认证爆破实战笔记

## 引言

本题为典型的HTTP基础认证（Basic Authentication）爆破场景。目标是网站受Basic认证保护，需要通过爆破获取正确的`用户名:密码`组合，并将其Base64编码后放入请求头，从而通过认证获取flag。

------

## 一、题目核心

- 

  **认证方式**：HTTP Basic 基础认证。

- 

  **凭证格式**：`用户名:密码`（例如 `admin:123456`）。

- 

  **传输方式**：将上述格式的字符串进行 **Base64编码**，放入HTTP请求头的 `Authorization`字段中。

- 

  **解题思路**：使用Burp Suite的Intruder模块，对`Authorization`头中的Base64串进行爆破，尝试不同的密码组合，通过响应特征（如长度）差异找出正确密码。

------

## 二、核心名词解析

1. 

   **`Authorization`(请求头)**

   - 

     **含义**：授权、认证。

   - 

     **作用**：在HTTP请求中，此头用于向服务器提交客户端的身份凭证。在Basic认证中，其值为 `Basic <Base64编码串>`。

2. 

   **`Basic`(认证模式)**

   - 

     **含义**：HTTP协议定义的一种最简单的认证模式。

   - 

     **格式**：`Authorization: Basic base64(“用户名:密码”)`。

3. 

   **编码 vs 加密**

   - 

     **Base64 (编码)**：一种用64个字符表示二进制数据的方法。**规则公开，可无损逆向解码**，**不具备保密性**，仅用于适配传输协议（如HTTP头只能放ASCII字符）。

   - 

     **加密**：需要通过密钥（或算法）将明文转换为密文，**无密钥无法（或极难）逆向**，主要用于保证数据的机密性。

   - 

     **本题关键**：Basic认证只是编码，**不是加密**。任何截获请求的人都可以轻松解码出明文的用户名和密码，因此**绝对不安全**，仅用于CTF题目或内部测试。

------

## 三、Burp Suite Intruder 爆破完整配置流程

### 第一步：捕获请求与设置攻击点

1. 

   浏览器配置代理（如127.0.0.1:8080），并访问目标网址。

2. 

   在弹出的Basic认证框中，随意输入信息（如用户名`admin`，密码`1`）。

3. 

   请求会被Burp Proxy拦截。将其发送到 **Intruder** 模块。

### 第二步：Positions 页面设置

1. 

   **攻击模式**：选择 **`Sniper`(狙击手模式)**。此模式适用于对单个位置（即Base64串）进行爆破。

2. 

   **清除与标记**：

   - 

     点击 `Clear §`按钮，清除所有默认的负载标记。

   - 

     在请求中找到 `Authorization: Basic`后面的那串Base64编码（例如 `YWRtaW46MQ==`）。

   - 

     **选中这串Base64编码**，然后点击 `Add §`按钮。此时该编码串前后会被添加 `§`符号标记为攻击位置。

### 第三步：Payloads 页面配置（核心步骤）

1. 

   **载荷类型**：选择 **`Custom iterator`(自定义迭代器)**。此功能允许我们将多个部分（账号、分隔符、密码）自动组合。

2. 

   **设置迭代器 (Payload Positions)**：

   - 

     将 **`Payload count`** 设置为 **`3`**。这表示我们将用3个部分来拼接。

   - 

     **Position 1 (第一部分 - 用户名)**：在下方列表中，为第一个位置添加一个固定值，例如 `admin`。

   - 

     **Position 2 (第二部分 - 分隔符)**：为第二个位置添加一个固定值，**必须是英文冒号 `:`**。

   - 

     **Position 3 (第三部分 - 密码字典)**：为第三个位置，点击 “Load...” 按钮，载入你的密码字典（如 `password.txt`），或手动添加常用密码（如 `123456`， `admin888`， `ctfshow`， `123123`等）。

   - 

     *效果预览*：最终生成的载荷将为 `admin`+ `:`+ `密码1`， `admin`+ `:`+ `密码2`……

3. 

   **载荷加工 (Payload Processing)**：

   - 

     这是最关键的一步！点击下方的 `Add`按钮。

   - 

     选择 `Encode`-> **`Base64-encode`**。

   - 

     *作用*：这会将我们拼接好的 `admin:密码`字符串，**自动进行Base64编码**，然后替换到请求的标记位置。**（必须开启）**

4. 

   **有效载荷编码 (Payload Encoding)**：

   - 

     **务必取消勾选** `URL-encode these characters`选项。

   - 

     *原因*：Base64编码串中可能包含 `+`， `/`， `=`等特殊字符。如果开启URL编码，这些字符会被转换成 `%2B`， `%2F`， `%3D`等形式，破坏了Base64格式，导致服务器无法识别，爆破必定失败。**（必须关闭）**

### 第四步：启动攻击与结果判断

1. 

   点击右上角的 `Start attack`按钮，开始爆破。

2. 

   攻击完成后，**主要观察 `Length`(响应长度) 这一列**。

3. 

   **判断依据**：绝大多数使用错误密码的请求，返回的响应页面（通常是401未授权或认证失败页）长度是相同或相近的。**找到那个 `Length`值与其他请求明显不同的响应**。

4. 

   这个长度不同的响应，对应的Payload（即正确的 `admin:密码`组合的Base64编码）就是正确的凭证。查看该请求的完整响应体，即可找到Flag。

------

## 四、关键功能与模式区分

| 功能/模式                           | 作用                                                         | 在本题的设置/选择                                  |
| ----------------------------------- | ------------------------------------------------------------ | -------------------------------------------------- |
| **Sniper (狙击手)**                 | 每次攻击只替换**一个**负载标记位置。                         | 攻击模式选此，因为我们只标记了Base64串这一个位置。 |
| **Custom iterator (自定义迭代器)**  | 允许用户定义多个字段，并自动将它们按顺序拼接成一个完整的字符串。 | 用于生成 `用户名:密码`的标准格式。                 |
| **Payload Processing (载荷加工)**   | 在Payload被放入请求**之前**，对其执行编码、哈希、添加前缀等操作。 | **必须添加 `Base64-encode`**，完成编码转换。       |
| **Payload Encoding (有效载荷编码)** | Burp在发送前，**自动**对Payload进行URL编码，以防止特殊字符破坏HTTP请求结构。 | **必须取消勾选**，以防止Base64串被二次编码而破坏。 |

------

## 五、实战避坑要点总结

1. 

   **格式必须正确**：`用户名`和 `密码`之间必须是**英文冒号 `:`**，不能用中文或其他符号。

2. 

   **编码开关要牢记**：

   - 

     **Payload Processing (Base64编码) 必须开**：因为我们构造的是明文，需要转换成服务端认识的Base64格式。

   - 

     **Payload Encoding (URL编码) 必须关**：因为Base64串是最终格式，不允许Burp再做任何修改。

3. 

   **结果看长度**：在无其他干扰的情况下，通过对比响应包的 **`Length`（长度）** 来识别成功请求是最直观有效的方法。成功认证后的页面（带Flag）与认证失败的页面（错误提示）大小通常不同。

------

## 总结

解决Basic认证爆破题的核心流程是：**捕获请求 -> 标记Base64位置 -> 使用Custom iterator拼接`user:pass`-> 用Payload Processing进行Base64编码 -> 关闭URL编码 -> 爆破并通过响应长度差异找出正确密码**。理解每个步骤的原理和配置原因，是灵活应对各类变形题目的关键。

# CTF脚本编程常见错误剖析与总结

*基于“MD5爆破”与“PHP伪随机数”两道CTF题目的反思*

## 一、 第一题：MD5 Token 爆破题错误全集

这道题集中暴露了我作为新手的典型问题：**思路清晰，但“笔”不听使唤**。

| 错误类型             | 错误代码/行为                                       | 错误原因分析                                                 | 正确写法                                                     | 后果与影响                                                   |
| -------------------- | --------------------------------------------------- | ------------------------------------------------------------ | ------------------------------------------------------------ | ------------------------------------------------------------ |
| **🔥 语法错误**       | `token_str(i)` (i为整数)                            | 混淆类型。`int`对象没有`.encode()`方法，只有`str`有。这是Python基础类型操作不熟。 | `token_str(str(i))`                                          | 直接报错：`AttributeError: 'int' object has no attribute 'encode'`，脚本无法运行。 |
| **✍️ 笔误错误**       | `mad5_hash[14]`                                     | 单纯的变量名拼写错误，将`md5`打成了`mad5`。                  | `md5_hash[14]`                                               | 运行时报`NameError: name 'mad5_hash' is not defined`。       |
| **🧮 运算逻辑错误**   | 将数字转为16进制字符串后**拼接**，再进行运算。      | **误解了题目原意**。题目PHP代码是`($x+$x+$x)/$x`，即**三个相同的数字相加**，结果为3。而我错误地理解为将数字的十六进制表示拼接。 | 直接进行整数运算：`x + x + x == 3 * x`恒成立，所以只需满足后续字符条件即可。 | 脚本逻辑与题目条件完全背离，导致即使运行也找不到任何有效解。 |
| **🧠 条件逻辑错误**   | `if md5_hash[1]!='3' or md5_hash[1]!=md5_hash[31]:` | **`or`逻辑使用错误**。这个条件永远为真，因为`md5_hash[1]`不可能同时等于`'3'`和另一个值。此处应表达“两者不都为3”。 | 应为：`if not (md5_hash[1] == '3' and md5_hash[31] == '3'):` 或等价写法：`if md5_hash[1] != '3' or md5_hash[31] != '3':`(这才是“不都为3”) | 过滤条件失效，导致本该被过滤掉的无效解进入后续判断，**但不会阻止找到正确答案**，只是增加了无效循环。 |
| **🚫 自定义限制错误** | 强制要求`md5_hash[1] == '3'`。                      | **过度理解题目，自己增加了不存在的要求**。题目只要求`md5_hash[1]`等于`md5_hash[31]`，并未要求它们等于`'3'`。 | 应将此条件移除，或改为`md5_hash[1] == md5_hash[31]`。        | 严重缩小了答案的搜索范围。如果正确答案的这两位是`'a'`或`'e'`等，我的脚本将永远找不到它，导致“跑不出答案”。 |
| **🚫 类型判断错误**   | `if md5_hash[1] == 0:`                              | 在Python中，字符串`'0'`和整数`0`是**不相等**的。这是基础的类型匹配问题。 | `if md5_hash[1] == '0':`                                     | 判断条件永远为假，脚本无法正确过滤掉包含字符`'0'`的MD5值，可能导致找到不符合题目“是数字”要求的错误答案。 |
| **🔁 效率/习惯错误**  | 找到答案后未立即终止循环。                          | 没有养成“找到即退出”的良好编程习惯。                         | 在打印正确答案后，添加`break`或`sys.exit()`。                | 脚本会无意义地继续遍历剩余数字，浪费计算资源。虽然不影响最终结果，但效率低下。 |

## 二、 第二题：PHP伪随机数(`mt_rand`)题错误分析

这道题的错误在于**对工具和原理的理解**，而非具体代码语法。

| 错误类型           | 错误代码/行为                                           | 错误原因分析                                                 | 正确写法/思路                                                | 后果与影响                                                   |
| ------------------ | ------------------------------------------------------- | ------------------------------------------------------------ | ------------------------------------------------------------ | ------------------------------------------------------------ |
| **🤔 原理理解错误** | 认为“伪随机数在相同种子下可能产生多种结果”。            | 对伪随机数生成器(PRNG)的核心特性理解不透彻。**对于确定性算法，相同的种子和调用序列必然产生完全相同的输出序列。** | 理解并牢记：**`mt_srand(种子)`后，`mt_rand()`的调用顺序固定，则结果唯一。** | 导致解题时产生不必要的困惑，怀疑自己脚本的正确性，或去寻找不存在的“多解”。 |
| **🛠️ 解题思路绕远** | 编写Python脚本，通过`subprocess`调用系统PHP来执行计算。 | 忽视了“用最适合的工具做最简单的事”的原则。题目环境是PHP，最直接、最无依赖的解法就是用PHP自身。 | 编写一个2行的PHP脚本： `<?php mt_srand(123456); echo mt_rand();` | 增加了代码复杂度（需要处理子进程、管道），引入了不必要的依赖（要求系统安装PHP且版本匹配），属于“杀鸡用牛刀”。 |

## 三、 核心问题总结与根因分析

1. 

   **“想当然”与马虎（主要痛点）**：

   - 

     **数字不转字符串**：思维还停留在“数字”上，没切换到编程所需的“字符串”操作。

   - 

     **拼写错误**：最不该犯的错误，但新手在专注逻辑时极易忽略。

   - 

     **类型对比错误**：对Python的强类型和`==`操作的行为不敏感。

2. 

   **逻辑运算符误用（CTF常见坑）**：

   - 

     混淆`and`和`or`的逻辑，尤其是德摩根定律应用不熟练（`not (A and B)`等价于 `(not A) or (not B)`）。这是CTF中构造条件判断时的**高频易错点**。

3. 

   **审题偏差与过度解读**：

   - 

     **误解运算**：将简单的算术运算复杂化为字符串/进制操作。

   - 

     **添加条件**：自己脑补题目中不存在的限制，这是解题大忌。必须严格忠于题目代码。

4. 

   **对底层原理模糊**：

   - 

     对伪随机数、编码函数等“黑盒”的内部确定性机制了解不深，导致解题思路摇摆。

## 五、 改正建议与行动清单

以后编写CTF脚本，遵循以下三步检查法：

1. 

   **预处理检查**：

   - 

     [ ] 所有送入哈希、编码函数的数据，确认其类型是否为`str`，必要时用`str()`转换。

   - 

     [ ] 仔细核对变量名、函数名是否有拼写错误。

2. 

   **逻辑与条件检查**：

   - 

     [ ] 将题目中的条件（尤其是PHP代码）逐字翻译，不增不减。

   - 

     [ ] 写出条件后，用`and`/`or`真值表或简单例子验证逻辑是否正确。

   - 

     [ ] 所有与字符串的比较，检查引号 `'`或 `"`。

3. 

   **原理与优化检查**：

   - 

     [ ] 理解题目涉及的函数（如`mt_rand`）的核心特性。

   - 

     [ ] 找到答案后，是否及时`break`？

   - 

     [ ] 是否有更直接、更简单的实现方式（如用PHP解PHP题）？
